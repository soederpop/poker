# Pokurr Wave 1-4 Execution Roadmap

> For Hermes: planning only. Do not implement from this file directly without opening a new execution pass.

Goal:
Create a concrete execution roadmap for Waves 1-4 that turns the existing roadshow/readiness ideas into an ordered plan focused on product usefulness before the regret-minimizer capstone.

Current context / assumptions:
- The repo already has a strong engine/runtime base: core game legality, tournament mechanics, local server runtime, house bots, and performance coverage are in place.
- Existing planning docs already define the high-level narrative:
  - `docs/planning/2026-03-28-public-roadshow-readiness.md`
  - `docs/planning/2026-03-28-offline-analysis-and-local-bot-product.md`
  - `docs/planning/2026-03-28-regret-minimizer-roadmap.md`
- `README.md` is already partially aligned around three pillars.
- CLI surface is currently concentrated in `commands/poker.ts`; standalone entrypoint is `src/cli.ts`.
- Source-mode command is `luca poker ...`; compiled mode command is `pokurr ...`.
- The user added `.claude/skills/luca-framework/`, so future implementation work should use Luca introspection (`luca describe`, `luca eval`, scaffolding tutorials) whenever command/runtime behavior is unclear.

Proposed wave mapping:
- Wave 1 = positioning + source/binary verification foundation
- Wave 2 = offline analysis productization
- Wave 3 = bot authoring/onboarding polish
- Wave 4 = arena/spectator/demo polish
- Regret-minimizer remains a later capstone wave after these four are stable

Why this order:
1. It preserves the existing planning principle: the repo should be useful before the research story.
2. It reduces demo risk by hardening the public entrypoints first.
3. It creates clean user funnels in order: understand -> run analysis -> build a bot -> run a local arena.

## Wave 1: Positioning, command parity, and verification matrix

Objective:
Make the product easy to explain and make the source-mode and compiled-mode command surfaces trustworthy.

Outcome required to exit Wave 1:
A new user can understand the product quickly, run the main commands in both modes, and get actionable failure messages when something breaks.

Primary deliverables:
1. One canonical product story across docs.
2. A source/binary verification matrix for core commands.
3. A public demo checklist that does not assume tribal knowledge.
4. A short Luca-aware maintainer workflow note for debugging command/runtime behavior.

Files likely to change:
- `README.md`
- `DEMO.md`
- `docs/planning/2026-03-28-public-roadshow-readiness.md`
- new: `docs/planning/2026-03-28-command-verification-matrix.md`
- new: `docs/planning/2026-03-28-demo-checklist.md`
- optional new maintainer doc: `docs/planning/2026-03-28-luca-debugging-notes.md`

Code/files to inspect during execution:
- `commands/poker.ts`
- `src/cli.ts`
- `container.ts`
- any binary packaging scripts in `scripts/`

Step-by-step plan:
1. Re-read `README.md`, `DEMO.md`, `commands/poker.ts`, and `src/cli.ts` together.
2. Create a command inventory table with these rows:
   - analyze equity
   - analyze range
   - analyze hand
   - sim
   - serve
   - seed
   - house status
   - new-agent
   - types
   - register
   - join
   - watch
   - dashboard
3. For each command, record:
   - source-mode example
   - compiled-mode example
   - required prerequisites
   - expected output or observable effect
   - likely failure modes
4. Add a verification matrix doc under `docs/planning/`.
5. Tighten `README.md` so the top 5 minutes of reading match the actual current command surface.
6. Tighten `DEMO.md` so it uses one canonical local setup flow with consistent ports and names.
7. Add a demo checklist for a presenter/operator:
   - before-show setup
   - smoke test sequence
   - fallback path if one command misbehaves
8. Add a small Luca debugging note for maintainers:
   - `luca describe`
   - `luca eval`
   - when to inspect source vs runtime container introspection
9. During implementation, run a smoke test matrix from source and from `./dist/pokurr`.
10. Capture any source/binary divergence as explicit issues in the plan doc or follow-on tasks.

Validation for Wave 1:
- `bun run build:wasm`
- `bun run test`
- `bun run compile`
- smoke test both modes for at least:
  - `analyze equity`
  - `analyze range`
  - `analyze hand`
  - `new-agent`
  - `types`
  - `serve --help` or a short-lived serve invocation if safe
- Manual doc review: README examples must match current flags/behavior.

Risks:
- `commands/poker.ts` is large, so behavior/docs can drift easily.
- Compiled-mode asset/path assumptions may still differ from source mode.

Definition of done:
- There is one canonical explanation of the product.
- There is a written source/binary verification matrix.
- Core commands have copy-pasteable examples that are known-good.

## Wave 2: Offline analysis as a standalone product

Objective:
Make the analysis commands useful enough that someone could adopt Pokurr just for studying poker hands/ranges/situations.

Outcome required to exit Wave 2:
A user can run analysis commands, understand the output, and use situation documents intentionally rather than accidentally.

Primary deliverables:
1. Improved readability and consistency for analysis output.
2. Better examples for hand/range/equity/situation workflows.
3. A canonical situation-doc convention.
4. Expanded study examples under `docs/situations/`.

Files likely to change:
- `commands/poker.ts`
- `README.md`
- `docs/offline-analysis.md`
- `docs/situations/README.md` if it exists, otherwise create it
- `docs/situations/*.md`
- tests touching analysis/situation behavior

Code/files to inspect during execution:
- `lib/situations.ts`
- `lib/cards.ts`
- analysis sections within `commands/poker.ts`
- `packages/pokurr-core/*` if output or semantics depend on engine APIs

Step-by-step plan:
1. Audit the current output of:
   - `analyze equity`
   - `analyze range`
   - `analyze hand`
   - `sim --situation ...`
2. Identify formatting inconsistencies in labels, card rendering, percentages, headers, and recommendations.
3. Decide whether analysis commands should expose a consistent “quick terminal summary” pattern.
4. Update `docs/offline-analysis.md` into a task-oriented guide:
   - compare two hands
   - compare two ranges
   - inspect a single hand in context
   - create and simulate a reusable situation doc
5. Define a canonical situation-doc convention:
   - frontmatter keys
   - naming style
   - folder usage
   - minimum examples required per street/spot type
6. Add 8-12 high-quality example situations spanning preflop/flop/turn/river.
7. If feasible without overreaching, add optional markdown export or report-style summaries for simulations; if not, document this as a follow-on after Wave 2.
8. Add or update tests around parsing and analysis output stability where practical.

Validation for Wave 2:
- `bun run test`
- manual smoke tests for all analysis commands
- verify every example in `docs/offline-analysis.md` is copy-pasteable
- verify at least 3 situation docs work end-to-end with `sim`

Risks:
- Output polish can sprawl if not scoped tightly.
- Too much narrative output could hurt CLI readability.

Definition of done:
- Offline analysis docs feel complete enough for a study user.
- Situation docs feel like a first-class asset format.
- Analysis output is readable and consistent across modes.

## Wave 3: Bot authoring and first-time user success

Objective:
Make it easy for a new user to scaffold a bot, understand `strategy.ts`, and get into a tight edit/run loop quickly.

Outcome required to exit Wave 3:
A fresh user can create, run, and slightly modify a bot in under 30 minutes without needing to understand Luca internals.

Primary deliverables:
1. A cleaner `new-agent` scaffold.
2. Reliable and well-explained `types` generation.
3. A stronger onboarding tutorial for writing/debugging a bot.
4. A “first winning bot” or equivalent beginner progression.

Files likely to change:
- `commands/poker.ts`
- `docs/writing-a-bot.md`
- `docs/strategy-globals-api.md`
- scaffolded file templates embedded in `commands/poker.ts`
- integration tests for agent/scaffold flows if added

Code/files to inspect during execution:
- `commands/poker.ts` functions for scaffold/readme/types content
- `features/strategy.ts`
- any decision-context types exported to agent authors
- generated scaffold output from `new-agent`

Step-by-step plan:
1. Generate a fresh sample bot and inspect the output as a first-time user.
2. Review the generated files for unnecessary complexity.
3. Trim the scaffold to the smallest useful set while preserving success.
4. Improve generated README text inside new agents:
   - one-file-first explanation
   - local loop steps
   - common debugging mistakes
5. Verify `types/pokurr.d.ts` is sufficient and points clearly to the reference docs.
6. Align `docs/strategy-globals-api.md` with the actual injected globals and `DecisionContext` shape.
7. Rewrite `docs/writing-a-bot.md` around a 15-30 minute success path:
   - scaffold
   - read one example decision
   - run locally
   - make one change
   - observe behavior
8. Add a “first winning bot” progression or a small checklist for iterative strategy improvement.
9. Consider adding a short “debugging your first bot” section covering invalid actions, missing types, and path mistakes.

Validation for Wave 3:
- `bun run test`
- create a fresh agent with `luca poker new-agent my-bot tag`
- run `types` in the generated agent
- verify the generated bot can join a local table following only the docs
- manual doc review for first-time-user clarity

Risks:
- Generated comments can become noisy and hurt comprehension.
- Docs can drift from the true runtime shape unless checked against code.
- This wave may expose opportunities to split scaffold/template logic out of `commands/poker.ts`.

Definition of done:
- Bot authoring feels approachable.
- Generated scaffolds are compact and understandable.
- Docs answer “what does decide() receive and return?” immediately.

## Wave 4: Local arena polish, spectator delight, and demo reliability

Objective:
Make the local multiplayer/demo loop reliable and fun enough for public demos, friend play, and repeatable showcase sessions.

Outcome required to exit Wave 4:
An operator can bring up a compelling local arena quickly, spectators can understand what is happening, and the workflow is resilient enough for live demos.

Primary deliverables:
1. A stable local operator flow for serve/register/join/watch/dashboard.
2. Clearer spectator/leaderboard/tournament guidance.
3. A repeatable tournament-night demo script.
4. Better smoke coverage for runtime happy paths.

Files likely to change:
- `commands/poker.ts`
- `servers/poker-server.ts`
- `DEMO.md`
- `docs/hosting-a-server.md`
- spectator/leaderboard web assets if needed
- `test/integration/runtime.test.ts`
- any demo helper docs under `docs/planning/`

Code/files to inspect during execution:
- `servers/poker-server.ts`
- `clients/poker-client.ts`
- `test/integration/runtime.test.ts`
- any web/spectator route files
- house actor definitions under `house/actors/`

Step-by-step plan:
1. Audit the exact local operator journey:
   - start server
   - inspect status
   - register bots
   - join a table
   - watch a table
   - inspect leaderboard/tournaments
2. Document every confusing point, missing hint, or weak error message.
3. Improve the CLI and/or docs so the happy path is obvious.
4. Review spectator and leaderboard surfaces for clarity during a live demo.
5. Tune house bot defaults or descriptions if the action is boring or hard to interpret.
6. Expand runtime integration coverage for the most important happy path sequence.
7. Produce a canonical demo script/checklist with:
   - setup
   - show flow
   - fallback branches
   - post-demo reset steps
8. Ensure local defaults are optimized for single-machine demos where possible.

Validation for Wave 4:
- `bun run test`
- targeted integration tests around runtime happy paths
- manual end-to-end demo dry run from a clean-ish terminal flow
- verify spectator URLs and leaderboard pages are reachable and legible

Risks:
- Live-demo issues often come from small operational assumptions, not core engine bugs.
- Web/spectator polish can sprawl if not bounded by operator needs.

Definition of done:
- A presenter can run the local demo from docs with minimal improvisation.
- Spectator output is understandable to a newcomer.
- The repo feels roadshow-ready for local-first usage.

## Cross-wave execution notes

Recommended sequencing constraints:
- Do not start major Wave 3 work until Wave 1 docs/verification are mostly settled.
- Do not start major Wave 4 polish until Wave 3’s authoring loop is understandable end-to-end.
- Keep the regret-minimizer as a later wave unless a small preparatory refactor clearly reduces future cost.

Recommended implementation style:
- Use small PR-sized chunks inside each wave.
- Prefer docs + smoke-test parity early, then deeper UX polish.
- When uncertain about Luca APIs or container behavior, prefer:
  1. `luca describe ...`
  2. `luca eval ...`
  3. source inspection
- Consider splitting oversized command/scaffold logic only when it directly helps a wave objective.

Suggested per-wave output artifacts:
- Wave 1: verification matrix + demo checklist
- Wave 2: polished offline-analysis docs + example situation set
- Wave 3: upgraded scaffold + onboarding docs
- Wave 4: operator script + strengthened integration coverage

## Risks, tradeoffs, and open questions

Risks:
- The single large `commands/poker.ts` file may slow down safe iteration.
- Docs can look polished while still drifting from actual source/binary behavior.
- Demo quality depends on operational sharpness, not just code quality.

Tradeoffs:
- Faster progress comes from documentation + smoke verification first.
- Deeper command refactors may be worthwhile, but only when they unblock clarity or testing.
- The regret-minimizer story is exciting, but moving it earlier would increase risk and fragment the product story.

Open questions to resolve during execution:
1. Should analysis commands eventually support `--json` consistently?
2. Should simulations emit markdown reports in Wave 2 or later?
3. Is `commands/poker.ts` still manageable for Wave 1-4 work, or should some logic be extracted first?
4. Which spectator/dashboard surfaces are essential for a roadshow vs nice-to-have?
